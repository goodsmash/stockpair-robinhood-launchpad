# V0.3.0 security design postmortem

## Status

The user reported that the website/contracts were hacked and users lost funds. The attack transaction hashes, affected deployed addresses, compromised frontend bundle, DNS/CDN logs, signing logs, and RPC traces were not supplied with this repair request. Therefore, this document distinguishes **confirmed design weaknesses** from an **unproven attack root cause**.

## Confirmed weaknesses in V0.3.0

1. The browser could consume a pool address from the indexer and approve/call it without independently proving the pool was registered by the configured factory.
2. Frontend trust was not anchored to an exact factory runtime hash and protocol-version constant before every write.
3. Normal pool functions accepted arbitrary recipients, allowing a malicious interface to redirect outputs while the user signed a superficially plausible transaction.
4. Creator allocations were immediately liquid and could be as high as 20%.
5. Initial LP locks could be as short as seven days.
6. Privileged configuration was controlled by a single immediate owner rather than an enforced on-chain delay.
7. A stock-token runtime hash alone did not exclude delegatecall proxies or dangerous privileged selector surfaces.
8. There was no protocol-level cap on swap input as a percentage of reserves.
9. There was no required oracle-denominated minimum initial stock value.
10. The browser allowed excessive slippage configurations, zero remove-liquidity minimums, and could leave residual allowances.
11. The indexer trusted forwarded client-IP headers by default and did not cap live-stream connections.
12. The CSP allowed broad network destinations instead of an explicit deployment allowlist.

Any one of these can increase loss severity. They do not, by themselves, prove which mechanism caused the reported theft.

## V0.4.0 remediation

- Direct chain verification of factory address, chain ID, runtime hash, protocol version, pool registry, token pair, launch record, launch-token issuer, stock code hash, oracle state, and emergency status before every browser write.
- Exact approval amounts and best-effort residual allowance revocation.
- Self-recipient enforcement inside the contracts.
- 5% reserve-based maximum swap input.
- 10% creator allocation cap with 90-day cliff and one-year linear vesting.
- One-year minimum initial LP lock.
- Mandatory 48-hour delayed administration with a seven-day execution window and guardian cancellation.
- Strict stock-asset bytecode policy rejecting delegatecall, callcode, selfdestruct, and listed privileged selector surfaces.
- Required oracle-denominated minimum initial stock liquidity.
- Fail-closed scanner and proxy evidence handling.
- Explicit trusted-proxy configuration, rate limits, global/per-IP SSE limits, and exact CORS allowlists.
- Restricted CSP and public administrative UI disabled by default.

## Limits

A maliciously replaced website can still display false information or induce signatures for unrelated external contracts. V0.4.0 prevents the legitimate StockPair contracts from paying to arbitrary recipients and makes the shipped browser verify its write targets, but users and operators must still secure DNS, CDN, CI, signing, RPC, and wallet infrastructure.
