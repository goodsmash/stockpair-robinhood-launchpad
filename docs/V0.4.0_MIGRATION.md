# V0.4.0 emergency migration

V0.4.0 is a new deployment. V0.3.0 contracts are immutable and are not patched by updating this repository.

## Required sequence

1. Pause and block the V0.3.0 deployment immediately where control remains available.
2. Remove the old frontend from service and publish exact old addresses as deprecated.
3. Snapshot old pools, LP holders, lock positions, creator balances, allowances, and affected transactions at a documented block.
4. Preserve self-directed incident exits where the old contracts safely support them. Do not create a migration transaction that grants an unlimited allowance to an unverified contract.
5. Independently audit V0.4.0 and the migration mechanism before handling real value.
6. Deploy V0.4.0 through Phase 1 (`DEPLOY_PHASE=schedule`), archive action calldata, wait at least 48 hours, then run Phase 2 (`DEPLOY_PHASE=execute`).
7. Have the intended multisig owner accept the two-step ownership transfer.
8. Run `npm run verify:deployment` against a second RPC with exact expected code hash and protocol version.
9. Deploy the indexer with production trading disabled and confirm factory trust, stock policy, feeds, and event history.
10. Build the frontend with exact trust anchors; publish and independently compare its content hash.
11. Run a limited testnet/canary program and independent penetration and contract testing.
12. Enable real-value activity only after security, legal/compliance, operations, and product/risk approval.

## User communication

State clearly that:

- the old deployment remains a separate on-chain system;
- a source update does not protect old approvals or recover stolen funds;
- users should revoke old spenders through a trusted interface;
- all new addresses and bytecode hashes must be verified through multiple independent channels;
- no migration or reimbursement is guaranteed unless separately approved and funded.
